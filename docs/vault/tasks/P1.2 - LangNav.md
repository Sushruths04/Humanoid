# P1.2 — LangNav (Humanoid-G1-LangNav-v0)

## Summary

CommandNav extended with a **frozen text encoder** (MiniLM sentence transformer). Instead of just a one-hot command, the policy also receives a 384-dimensional text embedding of the color name ("red marker", "blue marker", "green marker"). This is the first step toward genuine language conditioning.

**Result: 98.8% per-command success** (red/blue/green) — better than CommandNav's 94.5% because the richer embedding makes the conditioning clearer to the policy.

[Result doc](../../docs/results/p1_langnav.md)

---

## The Text Encoder

`programs/common/text_encoder.py`:
- Uses `sentence-transformers/paraphrase-MiniLM-L6-v2` (6-layer MiniLM)
- **Frozen during RL training** — we don't backprop through the encoder
- Embeddings are pre-computed offline and cached to JSON: `programs/common/cache/nav_command_embeddings.json`
- Cache is loaded at env reset time; no online encoder inference during training

```python
from programs.common.text_encoder import load_command_cache
cache = load_command_cache("programs/common/cache/nav_command_embeddings.json")
# cache = {"red marker": [0.12, -0.04, ...], "blue marker": [...], ...}
# 3 commands × 384 dim
```

**Why frozen?** RL reward signal is too noisy to fine-tune a text encoder. The frozen encoder gives stable embeddings; the RL policy learns to use them.

---

## Architecture Addition Over CommandNav

The only change from CommandNav: the observation includes an extra 384-dim embedding of the commanded color:

```python
def lang_nav_obs(env, num_markers=3, command_cache=None):
    onehot = target_id_to_onehot(env._nav_target_ids, num_markers)  # 3-dim
    rel = _commanded_target_xy(env) - _robot_xy(env)                # 2-dim
    emb = command_cache[env._nav_target_ids]                        # 384-dim
    return torch.cat([onehot, rel, emb], dim=-1)                    # 389-dim
```

All other components (reset event, steer event, reward) are identical to CommandNav.

---

## Why It Works Better Than CommandNav

The one-hot vector (2-dim) just says "marker 0 or marker 1." The text embedding (384-dim) carries richer semantic context — which makes it easier for the policy network to route the right behavior to the right command. This matters when `num_markers` grows; with 3+ targets, one-hot starts to be insufficient.
