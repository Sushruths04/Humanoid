CHAPTER 10

Compute & Workflow

Where the training actually runs, and how we move between machines in a single command.

The one rule

Lightning = the box you debug on. Modal = jobs that already work.
Short + parallel + compute-only → Modal. Long + interactive + rendering → Lightning AI.

When to use which

Two hard constraints

🎮

RT-core GPUs only

Isaac Sim needs L40S / L4 / A10. A100 / H100 won't work (no RT cores) — counter-intuitive but firm.

🖼️

Rendering ≠ compute

Cameras/video need Vulkan graphics, provisioned separately from CUDA. That path lives on Lightning.

Docker = switch machines in one command

The whole environment lives in one Docker image (humanoid-isaaclab), used by both Modal and Lightning so they never drift. Code is mounted at runtime, so editing code never needs a rebuild. Moving to a fresh machine is just:

# on a new box — one pull and you're ready
docker login ghcr.io
bash thesis/scripts/docker_image_portability.sh pull

Interview answer — "how do you manage compute?"
"One Docker image on a registry, used by both Modal (serverless, for short/parallel compute-only jobs) and Lightning (persistent, for interactive debugging and rendering). Code is mounted so there's no rebuild loop. I'm careful to use RT-core GPUs (L40S/L4) because Isaac Sim won't run on A100/H100, and I keep the camera/render path on Lightning where graphics is available."
