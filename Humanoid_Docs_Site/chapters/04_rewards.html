CHAPTER 04

Reward Design

The single most important — and most interview-asked — part. The reward is how you tell the robot what "good" means. Get it wrong and it cheats; get it right and it learns elegant behavior.

What a reward is

Every simulation step, the robot earns a number: high for desirable behavior, negative for bad behavior. PPO simply adjusts the policy to collect more reward over time. The art is designing the number so that "more reward" exactly equals "the behavior I want."

Reward hacking. Robots are lazy geniuses — they'll exploit any loophole. If you reward "height" only, it might jump and fall. So we use many small, balanced terms and watch the rollouts. (Our own G1 PRM once "learned" to read a hidden +/- marker instead of reasoning — a classic example of optimizing the metric, not the goal.)

The reward recipe

We build the total reward from four families. Each term is independent, so we can turn any one off to measure its effect (an ablation).

Example — robust locomotion (the part that's done)

Reward termEncourages
velocity trackingmove at the commanded speed/direction
uprightnesskeep the torso vertical
alive / episode lengthdon't fall (a fall ends the episode)
penalty torque & energymove efficiently, not violently
penalty action smoothnessno jittery, twitchy motion

Add rough terrain + domain randomization (Chapter 5) and this same reward produced our headline result: mean reward 22.8, 981 / 1000 steps upright. (Chapter 6.)

Example — the wakeboard start (the new task)

Here the reward literally encodes the 5 rules of a good wakeboard start, so the robot learns correct form, not just "stand up somehow."

Wakeboard ruleReward termWeight
Rise gradually (don't jump up)phase-gated height-progress + penalty if too fast+1.5
Board angle ≥ 10° (heels weighted)board_positive_angle+1.5
Arms straight, handle at hipsarms_straight + handle_at_hips+1.0 / +0.8
Don't pull against the ropepenalty pull_against_rope−1.0
Lean back slightly, knees softlean_back_moderate + knee_bend+0.7 / +0.8
Reach a stable ridesuccess_bonus (sparse)+50
Phase-gating, explained. "Height-progress" only counts after the first half-second. Why? So the robot can't get points by exploding upward immediately — it's forced to stay crouched and let the pull lift it, exactly like a real beginner is taught.
Interview answer — "walk me through your reward."
"It's a weighted sum of four families: task rewards (height, uprightness, velocity/glide, a sparse success bonus), biomechanics-shaping terms that encode domain rules — for the wakeboard, board angle ≥10°, straight arms, handle at the hips, gradual rise — an optional AMP style reward for natural motion, and penalties for energy, jerk, falling, and pulling against the rope. Each term is a separate manager term so I can ablate it. I phase-gate the height reward to prevent the 'stand up too fast' failure, and I watch rollouts to catch reward hacking."
